# Local Hybrid Search — 運用ガードレール

このファイルは `local_hybrid_search.ts` を操作する際の**ガードレール**（巻き戻し禁止ルール）を定義する。
変更には人間の明示的な合意が要る。

---

## 書き込みゲート

書き込みは agent からの明示的な呼び出しを前提とする。重要なのは「検索用の小型モデルが
裏で勝手に保存層を書き換える」のを防ぐこと。

- **書き込みは明示的な write gate を必ず通す**。`put()` / `addEvidence()` は agent の
  意図的なアクション（CLI コマンドや明示的に定義されたルール）からのみ呼ぶ。CLI の各
  サブコマンドはそれ自体が監査可能な write gate。
- 検索フローの副作用として保存層を**暗黙に書き換えない**（read と write を混ぜない）。
- agent 生成の主張は `epistemic`（`inference` / `hypothesis`）と provenance
  （`conversation` / `tool`）を必ず付け、後から「対話だけが根拠」を洗い出せるようにする。

## 鮮度と忘却

- **decay / 忘却曲線を実装しない。**
  鮮度は `last_confirmed_at` と明示的な `addEvidence()` による再確認で表現する。
- 時間経過だけで知識を自動的に stale 化しない。

## フィルタリングの方向

- **取り込み時にフィルタする**（retention でフィルタしない）。
  DB に入れるかどうかは入口で判断する。一度入ったものを自動削除しない。

## slug の不変性

- `slug` は概念の不変ハンドル。
  表示名・content は `put()` で更新できるが、slug 自体のリネームは行わない。
- Wiki リンクは slug を指す。slug を変えるとリンクが壊れる。
- **slug と id の意味は違う**: `slug` → 常に現行 **live**（`resolveSlug`）。
  `id` → その**特定版**（live/stale どちらも、`getPageById`。ページ行は削除されないため
  常に解決できる）。Wiki リンクは slug で、版参照（履歴・dump・検索結果の pageId）は id で。

## 埋め込みモデル

- **モデル名・次元・dtype を勝手に変えない。**
  現在: `sirasagi62/ruri-v3-310m-ONNX`（既定）/ 768 次元 / `q8`（env で上書き可）。
  次元の変更はすべてのベクトルの再計算を要求する（一方通行のドア）。
  変更する場合は設計メモの「再検討トリガー」に従い、明示的な合意を得る。

## ページとチャンク（真実と索引の分離）

- **`page.content`（全文 Markdown）が唯一の真実**。チャンクは `page` から導出される索引で、
  いつでも再生成できる。チャンク連結ではなく `page.content` から元 wiki を復元する。
- チャンクは検索・embedding の単位。更新時は内容ハッシュで未変更チャンクのベクトルを再利用し、
  旧版のチャンク（＋FTS/vec 行）は削除する（導出物なので破棄してよい）。

## チャンクの並び順

- **正準順序は `(page, ordinal)` 昇順**。`getChunks` は常にこの順で返し、これがページの読み順。
- **`hybridSearch` はスコア順**（読み順ではない）。読み順に戻すには `groupSearchResultsByPage`。
- **`ordinal` は版をまたいで比較不可**。`put` で更新すると章構成に応じて 0 から振り直される。

## ID 体系

- **`page.id` は UUIDv7**（時系列順 = `ORDER BY id` が作成順）。外部参照・dump ファイル名・
  ストア間マージに使う安定 ID。
- **`chunk.id` は整数 rowid のまま**（`fts_chunk` / `vec_chunk` の rowid が整数必須のため）。
  チャンクの安定外部 ID は **`chunk.uuid`（UUIDv7）**。
- `page_id` / `superseded_by` / `evidence.page_id` はすべて page の UUIDv7 を参照。
- **`chunk.id` は `AUTOINCREMENT` を外さない**。無いと chunk テーブルが空になった直後に SQLite が
  rowid を再利用し、古い（stale な）chunk id が別の新規チャンクへ静かに化ける事故になる
  （`getChunkById` 等が誤ったデータを返す）。

## dump（書き込みのみ）

- `dump` は**書き込み専用**。re-import（読み戻し）は未実装。追加するときは、素朴に
  「次の `---` 行を探す」パーサーではなく **YAML を正しく構文解析するリーダー**にする
  （本文中の水平線 `---` と front-matter の終端を混同しうるため）。
- ファイル名は `doc-{slug}-{id}.md`。slug の `/` は `__`、他のファイル名不可文字は `_` に置換する
  （`sanitizeSlugForFilename` を経由し、独自にエスケープを実装しない）。
- `--out` ディレクトリは自動作成してよい（モデルキャッシュの「`~/.cache` 外は自動作成しない」
  ガードとは**別方針**。dump 先はユーザーが明示指定した書き込み対象であり、typo で任意の場所に
  巨大ファイルをばらまくモデル DL とはリスクの性質が異なるため）。

## スキーマ不変条件

- 「1 slug に live は高々 1 件」を **`page`** で維持する（部分ユニークインデックス）。
- FTS (`fts_chunk`) / vec (`vec_chunk`) の `rowid` は **`chunk.id`（整数）** と同一であることを維持する。
- `page.content` を真実、チャンクを導出索引とする分離を壊さない。
- スキーマ変更時もこれらの不変条件を壊さない。

## 検索品質

- ベクトル量子化（int8 等）の導入前に、品質ベースラインとの比較を行う。
- 小型モデルへの切り替えは、実データで品質が落ちないことを測定してから。
